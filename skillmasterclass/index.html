01What is a Skill?

An Agent Skill is a folder containing a SKILL.md file (plus optional scripts, references and assets) that packages expertise + instructions + resources an AI agent loads on demand to perform a specific task.

The onboarding analogy. A prompt is shouting instructions at a new hire every single time. A skill is handing them a runbook, a checklist, and the exact scripts — once — so they do the job like your best tester, without you in the room.

📦 Packaged

Instructions, reference docs, and runnable scripts travel together in one folder.

🔎 Discovered

The agent reads only the description until your task matches — then it loads the rest. Progressive disclosure keeps context cheap.

♻️ Reusable

Version it, share it over git or a registry, compose it with other skills. Write once, use forever.

02Skill vs Prompt

A prompt is a message. A skill is a capability. The difference is reuse, discovery, and packaged resources.

PromptAgent Skill
LifespanOne messageReusable, versioned
DiscoveryYou must remember to type itAgent loads it when the task matches the description
ResourcesText onlyTemplates, checklists, runnable scripts, assets
SharingCopy-pastegit clone / registry / npm install
ConsistencyVaries per person & moodSame workflow & guardrails every run
CompositionHardSkills call other skills & scripts
Rule of thumb: if you have pasted the same instruction more than twice, it wants to be a skill.

03Anatomy of a Skill

Every skill is a folder. The only required file is SKILL.md — YAML frontmatter followed by a Markdown body.

SKILL.md 
---
name: test-plan-generator
description: >-
  Turn a JIRA ticket into a review-ready test plan. Use when someone says
  "write a test plan for JIRA-1234", pastes a story, or asks what to test.
  Fetches the ticket, finds gaps, drafts the plan, stops for human review.
license: MIT
metadata:
  author: TheTestingAcademy
  stlc-phase: Test Planning
---

# Test Plan Generator

## When to use
- A JIRA key or story is provided and testing must be planned.

## Workflow
1. Fetch the ticket (JIRA MCP tool, or scripts/fetch_jira.sh, or ask to paste).
2. Gap analysis against references/requirement-checklist.md.
3. Draft the plan into references/test-plan-template.md.
4. STOP at a Human Review Gate — never mark it final.

## Guardrails
- Never fabricate acceptance criteria; a missing AC is a finding.
- A human owns sign-off.
Write the description like a search query the future-you would type. Third person, concrete trigger phrases ("write a test plan for…", "find gaps in JIRA-123"). This one field decides whether the skill ever fires.

04Your First Skill — Test Plan Generator

The goal: hand it a JIRA ticket, and it fetches the ticket, reviews it, finds the missing pieces, fills the team's test-plan template, and then stops and asks a human to approve before going further.

Step 1 — Scaffold the folder

# Claude Code — personal skills live here
mkdir -p ~/.claude/skills/test-plan-generator/{references,scripts,copilot}
cd ~/.claude/skills/test-plan-generator
$EDITOR SKILL.md

Step 2 — Write SKILL.md

Use the frontmatter + body from §03. The four workflow steps are the skill: fetch → find gaps → draft → review gate.

Step 3 — Add the resources it reaches for

references/requirement-checklist.md

The gap-analysis checklist — functional, data, non-functional, clarity. Every ⚠️/❌ becomes a question for the ticket author.

references/test-plan-template.md

Scope · Gaps & Questions · Scenarios (P0/P1/P2) · Data & Env · Risks · Entry/Exit · Human Review Gate.

scripts/fetch_jira.sh 
#!/usr/bin/env bash
# Usage: ./fetch_jira.sh VOC-1234   (needs JIRA_BASE_URL, JIRA_EMAIL, JIRA_TOKEN)
set -euo pipefail
KEY="${1:?usage: fetch_jira.sh <ISSUE-KEY>}"
curl -sS -u "${JIRA_EMAIL}:${JIRA_TOKEN}" -H "Accept: application/json" \
  "${JIRA_BASE_URL}/rest/api/3/issue/${KEY}?fields=summary,description,acceptance,components,issuelinks" \
  | jq '{key,summary:.fields.summary,description:.fields.description}'
No JIRA access? The skill degrades gracefully: prefer a JIRA MCP tool → fall back to the script → fall back to "paste the ticket". It never invents ticket content.

Step 4 — The same skill in GitHub Copilot

Copilot doesn't read SKILL.md. Its equivalent is a prompt file — drop it at .github/prompts/test-plan.prompt.md and run /test-plan VOC-1234 in Copilot Chat.

.github/prompts/test-plan.prompt.md 
---
mode: agent
description: Generate a review-ready test plan from a JIRA ticket
---
You are a senior QA analyst. For ticket ${input:ticket}:
1. Fetch/read it (JIRA tool or ask me to paste — never invent content).
2. Gap analysis → a "Gaps & Questions for the author" table
   (missing ACs, edge cases, data, roles, perf/security/a11y, ambiguity).
3. Draft: Scope · Gaps · Scenarios (P0/P1/P2, each mapped to an AC) ·
   Data & Env · Risks · Entry/Exit.
4. STOP at a HUMAN REVIEW GATE — list assumptions & open questions,
   ask me to approve or edit before writing test cases or automation.

07Advanced Skills

⛓️ Chained skills

Analyzer → plan → scenarios → cases → automation → execution → bug → RCA → closure. Each skill's output is the next one's input; you review at every gate.

🔌 MCP-powered

Wire JIRA, GitHub, Playwright, or a DB as MCP tools so skills act on live systems instead of pasted text.

🧪 Self-reviewing

A skill whose last step spawns a sub-agent to adversarially verify its own output before showing you.

📊 Report-emitting

Skills that render results to an HTML report (see /showme) or a metrics dashboard for stakeholders.

Design principle for every advanced skill: more autonomy → more explicit human gates. The smarter the chain, the louder the "approve before I continue".

qaskills.sh — 450+ ready QA skills.

Don't hand-write every skill. qaskills.sh is a curated catalog of 450+ QA / SDET / automation skills — install any of them into Claude Code (or convert to a Copilot prompt) in one line.

450+

Skills across planning, design, API, mobile, performance, security, accessibility, CI/CD, reporting.

One-line install

Pick a skill, drop it into ~/.claude/skills/. No build step.

Curated

Each skill carries a description, guardrails, and — where useful — references & scripts.

Browse & install

  1. List the catalog. Fetch the index and skim by phase or tag. 
    # browse the catalog
curl -fsSL https://qaskills.sh/index.json | jq '.skills[] | {name, phase, tag}'
  2. Install one skill into Claude Code's skills dir. 
    curl -fsSL https://qaskills.sh/install.sh | sh -s -- bug-reporter
# → writes ~/.claude/skills/bug-reporter/SKILL.md (+ any refs/scripts)
  3. Or install a whole phase — grab every Test Design skill at once. 
    curl -fsSL https://qaskills.sh/install.sh | sh -s -- --phase "Test Design"
  4. Verify before you trust. Read the SKILL.md — check the description, the guardrails, and any script it ships — before letting the agent run it.
Safety first (this is a testing course, after all). Never pipe curl … | sh from a source you haven't read. Prefer curl -o install.sh, open it, then run. Pin a version/commit for reproducible installs.

Using a qaskills skill in GitHub Copilot

A catalog SKILL.md converts cleanly to a Copilot prompt file — the frontmatter body is the prompt.

# pull a skill and convert it to a Copilot prompt file
curl -fsSL https://qaskills.sh/raw/bug-reporter/SKILL.md \
  | sed '1,/^---$/d; /^---$/,/^---$/d' \
  > .github/prompts/bug-reporter.prompt.md
# then in Copilot Chat:  /bug-reporter
Workflow we teach: start from a qaskills.sh skill → tweak the description & guardrails for your team → commit it into your repo's .claude/skills/ so everyone shares the same behavior.
The most important part

Store & host your skills.

A skill on one laptop helps one person. A skill in a repo — or a registry — helps the whole team, forever. Two ways to store, from lazy to full-platform.

Option A · Lazy

GitHub repo of folders

Keep skills as .md files / folders in a git repo. Clone, submodule, or copy into .claude/skills/. Zero infra. Start here.

Option B · Full

React app + database registry

A web UI to browse skills, a DB of metadata, and a CLI/npm installer that pulls a skill straight from the GitHub-backed registry onto any machine.

Option A — GitHub as the source of truth

The whole "product" is a repo layout + a convention. This is genuinely enough for most teams.

tta-skills/                     # your GitHub repo
├─ skills/
│  ├─ README.md                 # the STLC map
│  ├─ 02-test-planning/test-plan-generator/SKILL.md
│  ├─ 06-defect-management/bug-reporter/SKILL.md
│  ├─ framework-packs/playwright-pack/…
│  └─ …
└─ registry.json                # name → path → version index

# install a skill on any machine
git clone https://github.com/TheTestingAcademy/tta-skills
cp -r tta-skills/skills/02-test-planning/test-plan-generator ~/.claude/skills/

# or wire the whole repo as a git submodule inside a project
git submodule add https://github.com/TheTestingAcademy/tta-skills .claude/skills-src
Storing as .md vs folders: a flat .md per skill is fine for prompt-only skills. The moment a skill needs a checklist, template, or script, promote it to a folder with SKILL.md + references/ + scripts/.

Option B — A React registry app

When you outgrow copy-paste: a small web app to browse, search, and one-click install skills — backed by a DB, sourced from the GitHub repo, installable via a CLI.

The install experience

# one command, from anywhere
npx tta-skills install test-plan-generator
# → resolves version from the registry
# → downloads the folder from the GitHub-backed registry
# → writes it into ~/.claude/skills/test-plan-generator/

npx tta-skills search "bug"        # find skills
npx tta-skills list --phase Closure
npx tta-skills update              # pull latest versions

The build prompt (copy → paste into Claude Code / Copilot)

Don't build the registry by hand. Hand this prompt to your agent and let it scaffold the whole thing.

prompt 
Build a "TTA Skills Registry" — a skill hosting + install platform.

STACK: React + Vite frontend, a small Node/Express (or Cloudflare Worker) API,
and SQLite (or D1) for skill metadata. GitHub is the source of truth for skill
content; the DB only holds searchable metadata + version pointers.

FEATURES
1. Ingest: a script reads every skills/**/SKILL.md in a GitHub repo (recursive —
   skills are nested under STLC phase folders), parses the
   YAML frontmatter (name, description, metadata.stlc-phase, version) and upserts
   a row per skill into the DB. Runs on a webhook / cron.
2. Web UI (React): searchable, filterable gallery of skills — card per skill with
   name, phase tag, description, version, and a "copy install command" button.
   Cream/warm theme, left sidebar filter by STLC phase. Detail page renders the
   SKILL.md.
3. Install API: GET /api/skills (list+search), GET /api/skills/:name (manifest),
   GET /api/skills/:name/download (zip of the folder pulled from GitHub raw).
4. CLI (`npx tta-skills`): `install `, `search `, `list [--phase]`,
   `update`. install resolves the manifest from the API, downloads the folder,
   and writes it into ~/.claude/skills// (respect --dir override).
5. Publish flow: `tta-skills publish ./skills/my-skill` validates the SKILL.md
   frontmatter, opens a PR to the GitHub repo.

CONSTRAINTS: no auth for v1 (public read); validate SKILL.md frontmatter with a
JSON schema before ingest; every install prints the skill's guardrails so the
user reads them before first run. Ship a README with deploy steps.
Lazy path that still works: skip the DB — the GitHub repo + a static registry.json committed alongside the skills is a complete "registry". The CLI reads the raw JSON from GitHub and downloads folders. Add the API/DB only when search over hundreds of skills gets slow.
